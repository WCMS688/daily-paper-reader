# A Fast Locality Simulator for GEMM Design-Space Exploration on Multi-Chiplet GPUs 

Euijun Chung, Hyesoon Kim 

Georgia Institute of Technology 

euijun@gatech.edu, hyesoon@cc.gatech.edu 

Abstract —Multi-chiplet GPUs split memory into local and remote HBM regions across a silicon interposer, and reducing the remote HBM traffic is crucial for the performance and energy efficiency of multi-chiplet GPUs. For general matrix multiplica-tion (GEMM), the dominant operator in large language models (LLMs), the resulting inter-chiplet traffic depends strongly on kernel choices such as operand layout, CTA traversal order, and data placement, and the optimal strategy to minimize remote ac-cesses is nontrivial. We present a fast, functional, tile-level locality simulator that models CTA scheduling, per-chiplet L2 caches, and local/remote HBM accesses to evaluate a full-size LLM GEMM configuration. Across representative LLM GEMMs, the simulator shows that remote traffic varies by up to 58 × across the design space for the same GEMM dimensions. Moreover, using the simulator as feedback, an agentic AI discovers that a 2D block-swizzle CTA traversal reduces remote traffic over the best 1D traversal by up to 5.1 × under round-robin placement, identifying CTA traversal order as a first-order, GEMM-dependent design knob for inter-chiplet traffic. 

I. I NTRODUCTION AND MOTIVATION 

Multi-chiplet GPUs scale compute and memory capacity past the reticle limit, but form non-uniform memory access domains within a package [1]–[3]. As shown in Fig. 1, a memory request is served either by a near HBM ( local ) or a far one across the interposer ( remote ); remote accesses add latency and energy consumption, and consume shared inter-chiplet bandwidth. Improving chiplet locality , the fraction of traffic served locally, is therefore central to chiplet-GPU efficiency [4]. GEMM dominates LLM training and inference, and its tiled execution exposes predictable CTA-to-operand affin-ity (Fig. 2), which locality-aware data placement and CTA scheduling exploit by co-locating each CTA with the data it consumes [5], [6]. This makes locality both important and, in principle, controllable. However, these control knobs span a large design space: per-operand row/column layout, CTA traversal order, and the placement of operand and output data across chiplets. Whether a given configuration improves locality depends on how effectively the L2 caches capture the resulting reuse, so the best choice is non-obvious and varies with the GEMM shape. For instance, for a single representative GEMM in LLM training, the remote HBM traffic spans from roughly 430 MB to 37 GB depending on the configuration. Finding the optimal configuration for these knobs is chal-lenging on today’s hardware and software. Cycle-level simulators can in principle measure this, but are impractically slow at LLM-GEMM sizes (e.g., one Qwen3 GEMM has (M, N, K ) = (196 ,608 , 16 ,384 , 2,048) )and couple the kernel and hardware models, so exploring a software knob means re-engineering the workload rather than changing a setting. To this end, we develop a lightweight functional simulator for design-space exploration of GEMM on multi-chiplet GPUs. II. L OCALITY SIMULATOR 

We model one MI300X-like package as four locality do-mains (chiplets), each with 76 compute units and a private 8 MB, 16-way L2 (128 B lines). The simulator is functional and trace-driven : each CTA computes one output tile with configurable size and issues strided operand accesses deter-mined by the GEMM shape and traversal order; accesses filter through the per-chiplet L2, and every L2 miss and output write is classified as local or remote based on the data-placement policy. Each CTA computes a output tile with A/B operand tiles held in shared memory, so per-CU occupancy is set by the shared-memory budget. CTAs within a wave advance through the K dimension in lockstep, issuing their strided A/B reads through the chiplet’s L2 at each K-step before any CTA proceeds.

The tunable knobs are (i) per-operand row/column layout; (ii) CTA traversal order in the kernel implementation, either 1D row- or column-major or a 2D block-swizzle; and (iii) data placement for the input and output matrices, either address round-robin at 4 KB/64 KB/2 MB granularity or contiguous per-chiplet. CTA scheduling then follows naturally from the placement of the output matrix. The outputs are L2 hit rate, HBM locality = local /(local +remote), and absolute inter-chiplet (remote) traffic. The simulator models data movement ,a first-order proxy for the latency and energy of remote access, rather than absolute timing. We do not model the memory-side last-level cache (e.g., AMD Infinity Cache), since it resides beyond the interposer crossing and therefore does not affect the local/remote classification of traffic, only the fraction that ultimately reaches DRAM. III. R ESULTS AND INSIGHTS 

We evaluate 12 representative feed-forward GEMMs from Qwen3-30B (mixture-of-experts) and Llama3.1-70B, in both the forward and backward pass: each FFN contributes a gate-up ( Up_ ) and down ( Down_ ) projection, each appearing as a forward (X) and two backward (dX, dW) GEMMs. In Fig. 3, 

row /col data interleaves the input matrices across chiplets by row/column, while row /col CTA interleaves the CTA grid, and hence the output tiles, across chiplets by row/column. In this evaluation, we use a tile size of (tm, t n, t k) = (128 , 128 , 32) 

and BF16 operands. We further assume that each chiplet’s private L2 caches remote as well as local accesses. The choice of data placement and CTA scheduling swings remote HBM traffic by up to 58 × for a single GEMM, and the best combination differs across GEMMs, so no fixed placement is optimal. Changing CTA traversal adds a further lever (Fig. 4). Each full-size 16K-token GEMM configuration simulates in 9–44 s (mean: 24 s) on a single CPU core, orders of magnitude cheaper than cycle-level methods at this scale, so a full design-space sweep is practical. We then used the simulator as feedback for an AI agent (Claude Code with Opus 4.7) tasked with reducing a GEMM’s remote traffic without changing the hardware or data place-ment. Exploring only the traversal knob, the agent adopted a 2D block-swizzle order (a standard L2-locality optimization in dense-GEMM kernel libraries such as CUTLASS and Triton) under hardware round-robin interleaving, where it reduces remote HBM traffic by up to 5.1 × (geometric mean 27.1% over the best 1D traversal) by improving per-chiplet L2 reuse (Fig. 4). The benefit is GEMM-dependent and neutral on some shapes, so the takeaway is not a single trick but an actionable insight delivered in seconds: CTA traversal order is a first-order, shape-dependent control for inter-chiplet traffic that AI and automated kernel generators must account for. More broadly, because the simulator exposes both hardware and software knobs in one fast model, it makes joint hard-ware/software optimization easier to explore. IV. I MPACT AND LIMITATIONS 

The simulator is a fast, AI-analyzable methodology for chiplet-GPU locality that isolates software knobs from the hardware model, enabling rapid and AI-driven design-space exploration. Its main limitation is that it is functional : it reports data movement, not absolute performance or power, and it assumes wave-synchronous (lockstep) CTA progress, which can slightly perturb the modeled L2 hit rate. Because the data footprint of one wave is smaller than the L2, we expect this effect on locality to be small. In short, the simulator turns chiplet-GPU GEMM locality into a fast, transparent design space that engineers and AI agents alike can explore in seconds. REFERENCES [1] A. Arunkumar, E. Bolotin, B.
